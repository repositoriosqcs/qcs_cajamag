    BUENAS PRACTICAS

    Introducción¶

    El proyecto de Data Warehouse para CAJAMAG se centra en optimizar y gestionar de manera avanzada el flujo de datos mediante procesos ETL (Extracción, Transformación y Carga). Esta solución tiene como objetivo centralizar y consolidar datos de múltiples fuentes, como bases de datos de beneficiarios, instituciones educativas, servicios sociales y datos transaccionales. De esta manera, se facilita la toma de decisiones estratégicas y operativas, garantizando la integridad, consistencia y disponibilidad de los datos.

    Para lograr un sistema robusto y escalable, se han implementado las mejores prácticas de programación "clean code" y técnicas de optimización, asegurando que el código cumpla con los requerimientos de procesamiento de datos, a la vez que sea claro, eficiente y mantenible. Cada bloque funcional del proyecto sigue una estructura modular que facilita su reutilización y actualización. Además, se ha adoptado un enfoque orientado a la calidad en el manejo de logs e informes, proporcionando información en tiempo real sobre el estado del proceso ETL.

    En resumen, este proyecto provee una plataforma sólida para el almacenamiento y consulta de datos, implementando buenas prácticas en el desarrollo de software que garantizan una infraestructura de datos confiable y de alta calidad que apoya el crecimiento y la eficiencia organizacional de CAJAMAG.

    1. Modularidad y Estructura de Código¶

    • Separación de Responsabilidades: Cada notebook (“DimDatosFijos”, “DimCalendarioMensual”, etc.) está diseñado para cumplir con un bloque funcional específico del proyecto. Las funciones en “Funciones.py” siguen el principio de Single Responsibility Principle (SRP), permitiendo que cada función gestione un aspecto específico del flujo ETL, como la conexión a la base de datos o la validación de datos.

    • Uso de Decoradores y Context Managers: Se utilizaron decoradores y context managers para optimizar la ejecución y centralizar acciones repetitivas, como la gestión de conexiones a bases de datos y el manejo de errores. Esto evita la repetición de código y mantiene la claridad en la implementación de funciones comunes.

    • Uso de Módulo Centralizado de Credenciales: Se centralizaron las credenciales de conexión en un módulo único, consumible desde todos los scripts del ETL, lo cual facilita la gestión y actualización de las credenciales de manera segura y evita la duplicación innecesaria. A continuación, se muestra un ejemplo de cómo se estructura este módulo: 

      # Diccionario con las configuraciones de cada base de datos
configuraciones_bd = {
    'minerva': {
        'usuario': '***',
        'contraseña': '**********',
        'host': '***.***.***.*',
        'base_de_datos': ''  # Sin base de datos especificada para Minerva
    },
    'dwh': {
        'usuario': '***',
        'contraseña': '**********',
        'host': '***.***.***.*',
        'base_de_datos': 'dwh'
    },
    'neith': {
        'usuario': '***',
        'contraseña': '**********',
        'host': '***.***.***.*',
        'base_de_datos': ''  # Sin base de datos especificada para Neith
    }
}
      Este módulo permite centralizar la configuración de conexión a diferentes bases de datos, lo que facilita el mantenimiento y mejora la seguridad al evitar tener credenciales distribuidas en múltiples scripts.

    2. Buenas Prácticas en Nombres y Documentación¶

    • Nombres Descriptivos: Las funciones y variables usan convenciones que mejoran la comprensión, como “guardar_en_dwh” o “obtener_conexion”. Esto mantiene la uniformidad en Python y facilita el mantenimiento y uso del código.

    • Docstrings en Funciones: Cada función en “Funciones.py” incluye un docstring detallado, alineado con el formato de Google Docstring o reStructuredText. Esto permite que cualquier desarrollador entienda rápidamente los objetivos de cada función, los parámetros necesarios y el valor de retorno.

    3. Optimización del Código¶

    • Ejecución Paralela: Utilizando “concurrent.futures”, se aplicaron técnicas de paralelización en operaciones intensivas de lectura y carga de datos, optimizando así la duración del proceso ETL sin comprometer la calidad del procesamiento.

    • Uso Eficiente de Pandas: Se utilizaron métodos vectorizados en lugar de bucles para mejorar el rendimiento en la transformación de datos. Esto garantiza un procesamiento eficiente y legible, siguiendo las mejores prácticas recomendadas.

    4. Logging Estructurado y Detallado¶

    • Niveles de Logging: Se configuró “logging” para capturar eventos a diferentes niveles (“INFO”, “DEBUG”, “ERROR”), permitiendo un seguimiento detallado de cada etapa del proceso ETL. Cada notebook registra mensajes en un archivo de log centralizado (“etl_proceso.log”).

    • Salida de Información al Cliente: Los registros de log incluyen detalles sobre la duración de cada proceso, la cantidad de registros procesados y el estado del procesamiento. Esta información proporciona una visibilidad clara del rendimiento y calidad del flujo de datos, estandarizando la forma en la que se muestra cada carga y descarga de información. Por ejemplo:

    • Conexión y Carga de Datos: 
      2024-10-27 09:32:35,975 - INFO - CONEXION A BASE MINERVA
2024-10-27 09:32:36,521 - INFO - Cargando Tra 
2024-10-27 09:41:15,061 - INFO - Cargada Tra, 282,958 registros finales obtenidos. --- 8.64 minutes ---

    • Descarga y Almacenamiento de Datos: 

      2024-10-27 09:53:25,218 - INFO - Intentando conexión a la base DWH...
2024-10-27 09:53:25,222 - INFO - Conexión a la base DWH establecida con éxito.
2024-10-27 09:53:26,237 - INFO - Almacenando tabla única en DWH como BD_Dim_Datos_Fijos
2024-10-27 09:56:32,425 - INFO - Tabla almacenada correctamente. 970,252 registros finales obtenidos.
2024-10-27 09:56:33,514 - INFO - ALMACENAMIENTO ---  --- 3.14 minutes ---
      Estos mensajes estandarizados permiten un seguimiento consistente y detallado del estado de cada carga y descarga, con una clara indicación del rendimiento temporal y los registros afectados.

    • Formato Personalizado de Logging: El formato de salida incluye %(asctime)s - %(name)s - %(levelname)s - %(message)s, garantizando que cada mensaje en el log sea claro respecto al origen, la fecha y la hora del evento.

    5. Manejo Robusto de Errores¶

    • Excepciones Específicas y Personalizadas: Se definieron excepciones específicas como “DBConnectionError” para gestionar problemas de conexión a la base de datos, proporcionando información detallada en los logs y facilitando la solución rápida de fallas.

    • Bloques “try-except” Acotados: Cada bloque “try” abarca solo las líneas necesarias, siguiendo las recomendaciones de Clean Code. Esto facilita la detección de errores y asegura que solo las secciones susceptibles están protegidas.

    6. Validación y Estandarización de Datos¶

    • Validación de Registros Duplicados: Mediante “StoreDuplicated”, se detectan y almacenan registros duplicados para su revisión posterior, garantizando la consistencia en el Data Warehouse.

    • Estandarización de Direcciones: Las direcciones se estandarizan con la función “estandarizar_direccion”, aplicando etiquetas de validación para asegurar la precisión en los datos geográficos.

    Conclusión¶

    Estas buenas prácticas de "clean code" y estructuración aseguran un flujo ETL eficiente, documentado y alineado con las mejores prácticas de Python. La salida clara al cliente en los logs y la estructuración modular de las funciones aseguran altos estándares de calidad y permiten la adaptación del proyecto para futuras expansiones o modificaciones. La combinación de modularidad, documentación detallada y optimización de código sienta las bases para un Data Warehouse escalable y sostenible que satisface las necesidades de CAJAMAG.
