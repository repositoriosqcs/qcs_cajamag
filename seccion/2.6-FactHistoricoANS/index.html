    2.6 Fact Historico ANS¶

    Introducción¶

    El proceso ETL para Fact_HistoricoANS se utiliza para extraer, transformar y cargar datos de tres fuentes de datos (subsi02, subsi15, y subsi23) en la base de datos subsidio, consolidando los registros relacionados con los historiales de afiliación de empresas, trabajadores y beneficiarios en el Data Warehouse (DWH). La extracción se realiza mediante una consulta UNION para combinar datos de las tres tablas, asignando una fuente específica a cada entidad según su tabla de origen y limitando los resultados ultimos 18 meses. Posteriormente, se aplican transformaciones, como la generación de una clave única de historial y la estandarización de nombres de columnas, para facilitar su análisis y almacenamiento. Finalmente, el DataFrame df_final se almacena en la tabla BD_Fact_HistoricoANS del DWH, con registros en el log para monitorear el rendimiento y la precisión del proceso.

    Consulta de Datos y Unión de Tablas¶

    En este código, se define una consulta SQL que combina datos de tres tablas (subsi02, subsi15 y subsi23) de la base de datos subsidio mediante la operación UNION. La consulta unifica registros de diferentes tipos de entidades (Empresa, Trabajador, y Beneficiario), asignándoles una fuente específica según la tabla de origen. Cada selección dentro de la UNION crea un campo periodo basado en la fecha (fecafi o fecpre), que concatena el año y el mes. También incluye los campos coddoc, Identificacion, fecafi, fecapr, Tipo y Fuente. En el caso de subsi23, se emplea una INNER JOIN con la tabla subsi22 para obtener información adicional de identificación, basada en el campo común codben. La condición WHERE restringe los resultados a los años 2022 y 2023 para todas las tablas.

    La consulta está diseñada para recuperar únicamente los registros de los últimos 18 periodos (meses) desde la fecha actual. Para lograrlo, En cada WHERE, se filtra el campo periodo utilizando la función DATE_SUB(CURRENT_DATE(), INTERVAL 18 MONTH), que calcula la fecha límite de hace 18 meses. Esta condición es dinámica y siempre asegura que solo se devuelvan registros dentro de este rango de 18 meses.

    En el caso de la tabla subsi23, se emplea una INNER JOIN con subsi22 para obtener información adicional de identificación, basada en el campo común codben.

    Finalmente, el logger registra un mensaje indicando la lectura de la consulta, lo que ayuda a monitorear la ejecución del proceso en tiempo real. La estructura generada en qr_structure estará lista para ejecutarse y almacenar los resultados en df_structure, permitiendo realizar análisis y procesamiento posteriores.

                select 
                    CONCAT(YEAR(fecafi), LPAD(MONTH(fecafi), 2, '0')) AS periodo,

                    fecafi,

                    'subsi02' Fuente
                from subsidio.subsi02
                where CONCAT(YEAR(fecafi), LPAD(MONTH(fecafi), 2, '0')) >= DATE_FORMAT(DATE_SUB(CURRENT_DATE(), INTERVAL 18 MONTH), '%Y%m')

                UNION

                select 
                    CONCAT(YEAR(s15.fecpre), LPAD(MONTH(s15.fecpre), 2, '0')) AS periodo,

                    s15.cedtra as Identifiacion, 
                    s15.fecpre AS fecafi,

                    'subsi15' Fuente

    Procesamiento de Datos y Transformación en df¶

    En este script, partiendo de df = df_structure['query'], se realiza un procesamiento de datos detallado que incluye la creación de nuevas columnas y la transformación de tipos de datos para estructurar la información en df_filtered. Primero, se crea una columna Validador que evalúa si fecafi es mayor que fecapr, asignando un valor de 0 o 1 según la comparación. Posteriormente, se introduce una columna tipide utilizando map para convertir los códigos en coddoc en identificadores numéricos específicos mediante coddoc_mapping, y se maneja cualquier valor no coincidente conservando su valor original.

    La columna KeyHistoricos se genera concatenando periodo, tipide, Identifiacion y las tres primeras letras de Tipo, formando una clave única para cada registro. Luego, se filtran las filas donde Validador es igual a 1, eliminando esta columna del DataFrame resultante df_filtered. Para mejorar la precisión en la manipulación de fechas, fecafi y fecapr se convierten al tipo datetime.

    El proceso de ETL se registra en el logger tanto en términos de la cantidad de registros finales obtenidos como del tiempo total de ejecución. El DataFrame final df_filtered se muestra para validar las transformaciones realizadas, y se guarda opcionalmente en el Data Warehouse para futuras consultas y análisis.

    # Asumiendo que df = df_structure['query']
# Crear la columna 'Validador' basado en la comparación entre 'fecafi' y 'fecapr'

df = df_structure['query']
df['Validador'] = df.apply(lambda row: 0 if row['fecafi'] > row['fecapr'] else 1, axis=1)

# Agregar columna 'tipide' usando map en lugar de apply
coddoc_mapping = {
    'CC': 1, 'CD': 8, 'CE': 4, 'NI': 7, 'NP': 5,
    'PA': 6, 'PE': 9, 'PT': 15, 'RC': 3, 'TI': 2
}

# Mapear directamente el valor de 'coddoc' a 'tipide'
df['tipide'] = df['coddoc'].map(coddoc_mapping).fillna(df['coddoc'])

# Agregar columna 'KeyHistoricos'
df['KeyHistoricos'] = df[['periodo', 'tipide', 'Identifiacion', 'Tipo']].agg(
    lambda x: f"{x['periodo']}{str(x['tipide'])}{x['Identifiacion']}{x['Tipo'][:3]}", axis=1
)

# Filtrar filas donde Validador es 1 y hacer una copia explícita
df_filtered = df[df['Validador'] == 1].copy()

# Eliminar columna 'Validador'
df_filtered.drop(columns=['Validador'], inplace=True)

# Cambiar el tipo de las columnas 'fecafi' y 'fecapr' a datetime
df_filtered['fecafi'] = pd.to_datetime(df_filtered['fecafi'], errors='coerce')
df_filtered['fecapr'] = pd.to_datetime(df_filtered['fecapr'], errors='coerce')

# Registrar el número de registros obtenidos

    Generación de ID Único y Selección de Columnas en df_final¶

    En este código, se crea la columna ID AFILIADO en df_filtered mediante la concatenación de los valores en coddoc y Identifiacion, ambos convertidos a texto, para formar un identificador único de afiliado. Seguidamente, el diccionario _names mapea los nombres originales de columnas a nombres más descriptivos y en mayúsculas, facilitando la estandarización y consistencia. La función convertir_columnas_mayusculas aplica estos cambios a df_filtered.

    Finalmente, se genera df_final seleccionando únicamente las columnas clave, incluyendo ID AFILIADO, PERIODO, CODIGO_DOCUMENTO, ID, FECHA_AFILIACION, FECHA_APROBACION, TIPO, FUENTE, TIPIDE y KEYHISTORICOS, lo que asegura que el DataFrame final esté estructurado de acuerdo con los requisitos para análisis o almacenamiento en la base de datos.
